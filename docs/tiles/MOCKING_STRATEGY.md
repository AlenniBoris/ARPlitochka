# Стратегия мокирования бэкенда

Для обеспечения автономной разработки и тестирования в проекте реализована система динамического мокирования ответов сервера.

## Принципы работы
1.  **Прозрачность**: Бизнес-логика (UseCase, Repository) не знает, откуда приходят данные — из реальной сети или из локального файла.
2.  **Динамическое переключение**: Переключение между реальным бэкендом и моками происходит во время выполнения (Runtime) с помощью флага `isMockEnabled`.
3.  **Локальные ресурсы**: Моковые данные (JSON) и медиа-файлы (текстуры) хранятся внутри модуля `:mock:tiles` в папке `assets`.

## Техническая реализация
*   **Ktor Interceptor**: В `HttpClient` встроен перехватчик запросов. Если `isMockEnabled == true`, запрос не уходит в сеть, а перехватывается.
*   **Mock Engine**: Перехватчик ищет соответствующий маршрут (Route) в модуле `:mock:tiles` и возвращает содержимое локального JSON-файла.
*   **Asset URLs**: Для текстур используются специальные URL-схемы:
    *   `file:///android_asset/...` для Android.
    *   Локальные пути для iOS.

## Использование
Для добавления нового мока:
1.  Создать JSON-файл в `:mock:tiles/src/main/assets/mock/...`.
2.  Зарегистрировать путь к нему в `TileAssets.kt`.
3.  Добавить маршрут в `TileRoutes.kt`.

Это позволяет фронтенд-разработчикам работать над UI и логикой, даже если API еще не готово или бэкенд недоступен.
