# AR Plitka Backend Roadmap

## Цель

Бэкенд нужен как отдельный серверный проект, который берет на себя бизнес-логику, хранение данных и тяжелые вычисления. Android-приложение остается AR-фронтом: оно определяет контур зоны, показывает пользователю визуализацию, дает выбрать плитку и угол поворота текстуры, а затем отправляет нормализованные данные на сервер.

Главная идея: мобильное приложение не должно считать оптимальную раскладку плитки, работу с обрезками, стоимость, проекты и каталог. Это зона ответственности бэкенда.

## Разделение Ответственности

### Android-приложение

- Определяет поверхность пола через ARCore.
- Позволяет пользователю выделить контур зоны точками.
- Показывает линии, точки, заливку и текстуру плитки.
- Позволяет повернуть текстуру/узор.
- Нормализует контур в 2D-координаты в метрах.
- Отправляет на сервер контур, выбранную плитку, угол поворота и параметры раскладки.
- Получает результат расчета и отображает его пользователю.

### Бэкенд

- Хранит пользователей, проекты, зоны, каталог плитки и текстуры.
- Выполняет расчет площади, периметра и количества плитки.
- Строит схему раскладки плитки внутри полигона.
- Учитывает разные размеры плитки в рамках одной коллекции.
- Позже оптимизирует использование обрезков.
- Рассчитывает стоимость.
- Генерирует отчет, смету или PDF.
- Возвращает Android-приложению готовую расчетную модель.

## Рекомендуемый Стек

Так как Android-приложение уже написано на Kotlin, для бэкенда логично использовать Kotlin.

### Базовый вариант

- Kotlin
- Ktor
- PostgreSQL
- Exposed или jOOQ
- Flyway для миграций БД
- Koin или Kodein для DI, либо ручная сборка зависимостей на старте
- kotlinx.serialization для DTO
- Docker и Docker Compose

### Альтернатива

- Kotlin
- Spring Boot
- PostgreSQL
- Spring Data JDBC или jOOQ
- Flyway
- Docker

### Рекомендация

Для этого проекта лучше начать с Ktor. Он легче, проще для отдельного API-сервиса и хорошо подходит под Clean Architecture, где HTTP-слой не должен управлять доменной логикой.

## Архитектурный Подход

Бэкенд стоит строить по Clean Architecture / Hexagonal Architecture.

Главное правило: расчет раскладки плитки не должен жить в HTTP-контроллерах, ORM-моделях или Android-контрактах. Нужно выделить независимое расчетное ядро.

Пример структуры будущего backend-проекта:

```text
ar-plitka-backend/
  app/
    src/main/kotlin/
      config/
      di/
      routing/
      serialization/
      database/
    src/main/resources/
      application.conf

  domain/
    src/main/kotlin/
      catalog/
      project/
      surface/
      layout/

  layout-engine/
    src/main/kotlin/
      geometry/
      tiling/
      optimization/

  infrastructure/
    src/main/kotlin/
      database/
      storage/
      repositories/

  docker-compose.yml
  build.gradle.kts
  settings.gradle.kts
```

## Ключевые Модули

### app

Точка входа сервера. Здесь живут Ktor, роутинг, настройка JSON, CORS, логирование, DI и запуск приложения.

Этот слой принимает HTTP-запросы, валидирует DTO и вызывает use case. Он не должен считать плитку сам.

### domain

Доменная модель проекта:

- `Project`
- `Surface`
- `TileCollection`
- `TileVariant`
- `TilePattern`
- `LayoutRequest`
- `LayoutResult`

Здесь же живут use cases:

- `CreateProjectUseCase`
- `SaveSurfaceUseCase`
- `CalculateLayoutUseCase`
- `GetTileCatalogUseCase`

### layout-engine

Самая важная часть системы. Это расчетное ядро, которое не знает ничего про HTTP, БД и Android.

Оно принимает:

- полигон зоны;
- размеры плитки;
- угол поворота узора;
- стратегию раскладки;
- параметры допуска.

И возвращает:

- площадь;
- периметр;
- количество целых плиток;
- количество резаных плиток;
- список размещенных плиток;
- список обрезков;
- процент отходов;
- расчетную схему.

### infrastructure

Здесь находятся реализации репозиториев, работа с PostgreSQL, файловым хранилищем, внешними сервисами и миграциями.

## Нормализация Данных с Android

На сервер не нужно отправлять ARCore `Anchor`, `Pose`, camera matrix или координаты устройства. Серверу нужна стабильная 2D-геометрия.

Android должен преобразовать выделенный контур в координаты относительно первой точки или центра зоны:

```json
{
  "points": [
    { "x": 0.0, "y": 0.0 },
    { "x": 2.4, "y": 0.0 },
    { "x": 2.4, "y": 1.7 },
    { "x": 0.0, "y": 1.7 }
  ],
  "unit": "meters"
}
```

В Android это будут координаты из AR-плоскости:

- AR `x` становится backend `x`;
- AR `z` становится backend `y`;
- высота `y` из ARCore серверу для расчета раскладки не нужна.

```json
        { "id": "large", "widthMm": 780, "heightMm": 1040 }
      ]
    }
```

```http
POST /api/v1/layout/calculate
```

Запрос:


## Этапы Разработки

### Этап 1. Создать отдельный backend-проект

Создать рядом с Android-проектом отдельный репозиторий или отдельную директорию:

```text
ar-plitka-backend/
```

На старте лучше отдельный репозиторий. Так проще деплоить, версионировать и развивать сервер независимо от Android.

Что сделать:

- настроить Kotlin + Ktor;
- добавить health endpoint `GET /health`;
- настроить JSON-сериализацию;
- добавить Dockerfile;
- добавить Docker Compose с PostgreSQL.

### Этап 2. Описать доменные модели

Сначала без БД и без сложной геометрии.

Нужно описать:

- `Point2D`;
- `Polygon`;
- `TileCollection`;
- `TileVariant`;
- `Texture`;
- `LayoutRequest`;
- `LayoutResult`.

На этом этапе важно зафиксировать API-контракт между Android и backend.

### Этап 3. Реализовать простой layout-engine

Первая версия должна уметь:

- принимать полигон;
- считать площадь;
- считать периметр;
- накладывать прямоугольную сетку плитки с учетом угла поворота;
- понимать, какие плитки полностью внутри зоны;
- понимать, какие плитки пересекают границу и требуют резки;
- возвращать базовую статистику.

На этом этапе не нужно пытаться идеально переиспользовать обрезки.

### Этап 4. Подключить БД

Добавить PostgreSQL и миграции.

Минимальные таблицы:

- users;
- projects;
- surfaces;
- tile_collections;
- tile_textures;
- tile_variants;
- layout_calculations.

На старте авторизацию можно не делать, если приложение еще прототип. Можно хранить проекты по локальному `deviceId` или временно работать без пользователей.

### Этап 5. Подключить Android к backend

В Android добавить сетевой слой:

- Retrofit или Ktor Client;
- DTO для запросов и ответов;
- repository;
- use case `CalculateLayoutUseCase`;
- UI-состояния `Uploading`, `Calculating`, `Success`, `Error`.

При нажатии `OK` Android должен:

- взять точки контура;
- преобразовать AR `x/z` в 2D-полигон;
- добавить выбранную плитку;
- добавить угол поворота текстуры;
- отправить запрос на сервер;
- получить расчет;
- показать результат.

### Этап 6. Сохранять проекты

Добавить возможность:

- создать проект;
- сохранить выделенную зону;
- сохранить выбранную плитку;
- сохранить расчет;
- открыть проект повторно.

### Этап 7. Улучшить расчет раскладки

После MVP можно развивать расчетное ядро:

- поддержать несколько размеров плитки в одной коллекции;
- добавить разные паттерны укладки;
- учитывать направление укладки;
- оптимизировать количество резаных плиток;
- переиспользовать обрезки;
- считать отходы;
- сравнивать несколько стратегий и выбирать лучшую.

### Этап 8. Генерировать отчет

Добавить:

- PDF-смету;
- список плитки;
- схему укладки;
- расчет стоимости;
- процент отходов;
- изображение итоговой раскладки.

## Как Коннектить Android и Backend

### Android-side слои

В Android стоит добавить отдельный data/network слой:

```text
features/floor-detection/
  data/
    remote/
      dto/
      FloorLayoutApi.kt
      FloorLayoutRemoteDataSource.kt
    repository/
      LayoutRepositoryImpl.kt
  domain/
    repository/
      LayoutRepository.kt
    usecase/
      CalculateLayoutUseCase.kt
```

### Поток данных

```text
FloorArScreen
  -> FloorArViewModel
  -> CalculateLayoutUseCase
  -> LayoutRepository
  -> FloorLayoutApi
  -> Backend
```

Ответ идет обратно тем же путем и попадает в `FloorUiState`.

## Важные Технические Решения

### Координаты

Сервер работает только в 2D. Android отправляет точки в метрах.

Хороший вариант:

- первая точка контура становится `(0, 0)`;
- остальные точки считаются относительно нее;
- направление полигона сохраняется;
- угол текстуры отправляется отдельно.

### Версионирование API

Сразу использовать префикс:

```text
/api/v1
```

Это позволит позже менять контракт без поломки старых версий мобильного приложения.

### Синхронный и асинхронный расчет

Для MVP можно делать синхронно:

```text
POST /api/v1/layout/calculate
```

Позже, если расчет станет тяжелым:

```text
POST /api/v1/layout-jobs
GET /api/v1/layout-jobs/{id}
```

### Тестирование

Обязательно тестировать `layout-engine` отдельно от сервера.

Нужны тесты на:

- площадь полигона;
- периметр;
- поворот координат;
- пересечение плитки и полигона;
- расчет прямоугольной зоны;
- расчет зоны сложной формы;
- стабильность результата при разных углах поворота.

## Рекомендуемый Порядок Ближайших Действий

1. Создать отдельный backend-проект на Kotlin + Ktor.
2. Добавить `/health`.
3. Описать DTO для `LayoutCalculateRequest` и `LayoutCalculateResponse`.
4. Создать независимый модуль `layout-engine`.
5. Реализовать расчет площади и периметра.
6. Реализовать простую сеточную раскладку одной плиткой.
7. Написать unit-тесты на расчетное ядро.
8. Подключить Android к локальному backend через Retrofit или Ktor Client.
9. После нажатия `OK` отправлять контур и угол поворота на сервер.
10. Вернуть на Android результат расчета и показать пользователю.

## Что Не Делать Сразу

- Не начинать с авторизации и личного кабинета.
- Не пытаться сразу идеально оптимизировать обрезки.
- Не хранить ARCore-координаты сервера как главный формат.
- Не смешивать HTTP-контроллеры и расчетную геометрию.
- Не делать layout-engine зависимым от PostgreSQL.
- Не усложнять API до появления реальных сценариев.

## Итоговая Стратегия

Сначала строим простой, надежный backend-контур:

```text
Android contour -> Backend API -> Layout Engine -> Result -> Android UI
```

После этого постепенно усиливаем расчетное ядро: разные размеры плитки, паттерны, обрезки, оптимизация, сохранение проектов и отчеты.

Самое важное на старте - правильно отделить расчетную геометрию от Android, HTTP и базы данных. Тогда проект можно будет спокойно развивать без переписывания всей системы.
