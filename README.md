# AR Plitka Android

Android-фронт приложения AR Plitka. Приложение использует камеру, ARCore и SceneView, чтобы находить пол, позволять пользователю разметить контур зоны и примерять текстуры плитки в реальном масштабе.

Сейчас проект развивается как AR-клиент: он отвечает за камеру, AR-сессию, разметку, визуализацию, выбор плитки и подготовку пользовательского результата. Тяжелые расчеты раскладки, проекты, каталог, база данных и оптимизация обрезков должны быть вынесены в backend. План backend-части описан в `BACKEND_ROADMAP.md`.

## Что Уже Реализовано

- Одноэкранный AR-сценарий на Jetpack Compose.
- Проверка разрешения камеры и поддержки ARCore.
- Поиск горизонтальных плоскостей пола через ARCore.
- Reticle в центре экрана и hit-test по текущей точке наведения.
- Добавление точек контура с привязкой к ARCore `Anchor`.
- Соединение точек линиями, подписи длин сегментов и динамическая линия к reticle.
- Замыкание полигона при наведении на первую точку и финализация через `OK`.
- До финализации замкнутый контур показывается обычной заливкой.
- После финализации включается текстура плитки, повторяемая в физическом масштабе `780x1040 мм`.
- Поворот узора плитки внутри фиксированной секции с шагом `45°`.
- Переключение между несколькими типами плитки без удаления предыдущих текстур.
- Debug-панель только в debug-сборке.

## Пользовательский Сценарий

1. Пользователь открывает приложение и дает разрешение на камеру.
2. Приложение проверяет доступность ARCore и запускает AR-сцену.
3. Пользователь наводит камеру на пол, пока приложение ищет горизонтальную поверхность.
4. Reticle показывает текущую точку попадания в центр экрана.
5. Пользователь ставит точки по контуру нужной зоны.
6. Приложение соединяет точки линиями, показывает длины и динамическую линию к reticle.
7. Когда reticle подводится к первой точке, полигон считается замкнутым.
8. Нажатие `OK` финализирует секцию.
9. После финализации ARCore plane renderer скрывается, а внутри контура появляется плиточная текстура.
10. Пользователь может вращать узор плитки на `45°` и переключать тип плитки.
11. Сброс очищает секцию и возвращает поиск/разметку в начальное состояние.

## Gradle-Модули

```text
AR Plitka
├── app
├── features
│   └── floor-detection
└── shared
    ├── ar
    └── ui
```

### `:app`

Application-модуль. Содержит точку входа, Hilt application class, runtime-проверку камеры, проверку ARCore и подключение основного feature-модуля.

Подробнее: `app/README.md`.

### `:features:floor-detection`

Главная фича приложения. Содержит AR-экран, ViewModel, use case, repository, доменные модели, управление точками, финализацию секции, выбор текстуры и визуализацию плитки.

Подробнее: `features/floor-detection/README.md`.

### `:shared:ar`

Общие ARCore-расширения: фильтрация горизонтальных плоскостей, расчет площади ARCore plane и hit-test из центра экрана.

Подробнее: `shared/ar/README.md`.

### `:shared:ui`

Переиспользуемые Compose-компоненты: reticle, статусная панель, blocking screen, debug panel и `UiText`.

Подробнее: `shared/ui/README.md`.

## Архитектура

Проект разделен на application-модуль, feature-модуль и shared-модули. Внутри `:features:floor-detection` используется близкий к Clean Architecture подход:

- `presentation` управляет UI, `ARSceneView`, SceneView nodes и `FloorUiState`;
- `domain` содержит модели, enum-состояния, repository contract и use case;
- `data` обрабатывает ARCore `Session`, `Frame`, plane detection и hit-test;
- `di` связывает интерфейсы и реализации через Hilt.

Важное архитектурное правило: repository не управляет UI-состоянием. Он возвращает результат обработки AR-кадра, а `FloorArViewModel` собирает из него состояние экрана.

## AR И Стабилизация

Точки контура создаются из ARCore `HitResult` и закрепляются через `Anchor`. Это важно, потому что ARCore может уточнять world mapping во время работы приложения и после перехода в `onStop`/возврата. Поэтому `ArPoint` хранит не только `Anchor`, но и актуальный `Pose`, а ViewModel обновляет pose всех точек на каждом AR-кадре.

Так Compose получает новое состояние и перерисовывает контур по актуальным anchor pose, не полагаясь на getter, который UI мог бы не заметить.

## Текстуры Плитки

Текстуры лежат в `features/floor-detection/src/main/assets/textures/`.

Сейчас доступны:

- `paving_stones.png` - первый вариант плитки;
- `paving_stones_v2.png` - второй вариант плитки.

Тип плитки хранится в enum `TileType`, а выбранный тип - в `FloorUiState.selectedTileType`. Физический размер обеих текстур одинаковый: `780x1040 мм`.

Поворот хранится отдельно через enum `TextureRotation`: `DEGREES_0`, `DEGREES_45`, `DEGREES_90`, `DEGREES_135`. Значение градусов маппится на UI/rendering уровне, чтобы доменная модель оставалась удобной для будущей отправки на backend.

Для корректного `0°` секция строит собственную локальную систему координат: ось `X` направлена вдоль самой длинной стороны полигона, ось `Y` - поперек. Поэтому базовая раскладка плитки ложится длинными рядами вдоль длинной стороны выделенной зоны, а повороты `45/90/135°` применяются уже относительно этого нуля.

## Сборка И Запуск

Собрать debug-версию:

```powershell
./gradlew :app:assembleDebug
```

Установить на подключенное устройство:

```powershell
./gradlew :app:installDebug
```

Запустить установленное приложение:

```powershell
C:\Users\User\AppData\Local\Android\Sdk\platform-tools\adb.exe shell am start -n com.example.arplitka/com.example.arplitka.MainActivity
```

Собрать, установить и запустить одной командой в PowerShell:

```powershell
./gradlew :app:installDebug; if ($?) { C:\Users\User\AppData\Local\Android\Sdk\platform-tools\adb.exe shell am start -n com.example.arplitka/com.example.arplitka.MainActivity }
```

## Требования К Устройству

- Android 8.0+ (`minSdk = 26`);
- обычная камера;
- AR camera capability;
- поддержка ARCore;
- установленный или доступный Google Play Services for AR.

## Ближайшее Развитие

Android-часть логично развивать в сторону более удобного фронта примерки:

- выбор плитки из нормального UI-каталога;
- сохранение выбранного типа плитки и угла поворота;
- экспорт полигона, `TileType`, `TextureRotation` и размеров в backend DTO;
- улучшение поиска пола в тенях и сложном освещении;
- ограничение зоны пола плинтусами или другими границами помещения.
