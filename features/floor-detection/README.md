# Module `:features:floor-detection`

Feature-модуль основной AR-функциональности: поиск пола, разметка контура зоны, финализация секции и визуальная примерка плитки.

Это главный фронтовый модуль текущего приложения. Он содержит пользовательский AR-сценарий, состояние экрана, доменные модели разметки, обработку AR-кадров и SceneView-визуализацию.

## Назначение

Модуль отвечает за полный сценарий выделения зоны на полу:

- настройка AR-сессии и горизонтального plane detection;
- hit-test из центра экрана;
- отображение reticle, статусов и инструкций;
- добавление точек контура;
- стабилизация точек через ARCore `Anchor`;
- соединение точек линиями;
- отображение подписей длин сегментов;
- snap к первой точке и замыкание полигона;
- финализация секции через `OK`;
- отображение обычной заливки до финализации;
- отображение текстуры плитки после финализации;
- переключение типа плитки;
- вращение узора плитки внутри фиксированного контура;
- сброс выделенной секции.

## Архитектурные Слои

### `presentation`

Слой UI и состояния экрана.

`FloorArScreen` отвечает за Compose UI, `ARSceneView`, SceneView nodes, материалы и визуальные элементы: точки, линии, подписи, заливку, reticle и кнопки управления.

Вычисления, не являющиеся UI-разметкой, вынесены в отдельные файлы рядом с экраном:

- `FloorArGeometryUtils` - геометрия секции, локальные оси, bounds, линии и углы;
- `FloorArTextureUtils` - генерация bitmap-паттерна плитки и маппинг `TextureRotation` в градусы;
- `FloorArLabelUtils` - bitmap-подписи длин и форматирование метров;
- `FloorArRenderConfig` - render-константы размеров, толщин, offsets и физического размера текстуры.

`FloorArViewModel` управляет `FloorUiState` через `MutableStateFlow`, обрабатывает результаты AR-кадров, добавляет и удаляет точки, обновляет anchor pose, замыкает полигон, финализирует секцию, переключает плитку и вращает текстуру.

### `domain`

Доменный слой задает модели и контракты фичи:

- `FloorUiState` - полное состояние AR-экрана;
- `ArPoint` - точка пользовательского контура с `Anchor`, актуальным `Pose` и `id`;
- `ArFrameResult` - результат обработки одного AR-кадра;
- `FloorDetectionState` - состояние поиска пола;
- `TextureRotation` - дискретный enum поворота узора;
- `TileType` - enum доступных текстур плитки;
- `IFloorDetectorRepository` - контракт источника AR-данных;
- `ProcessArFrameUseCase` - use case обработки кадра.

Доменный слой не должен зависеть от конкретной реализации repository или SceneView rendering.

### `data`

`FloorDetectorRepositoryImpl` получает `Session`, `Frame` и размер viewport, фильтрует горизонтальные плоскости, делает hit-test по центру экрана и возвращает `ArFrameResult`.

Repository не управляет UI-состоянием. Это принципиально: repository отвечает за факты из ARCore, а ViewModel решает, как эти факты меняют экран.

### `di`

`FloorDetectionModule` связывает `IFloorDetectorRepository` с `FloorDetectorRepositoryImpl` в `ViewModelComponent`.

## AR-Сценарий

1. `ARSceneView` создает AR-сессию.
2. Конфигурация включает горизонтальный поиск плоскостей:
   - `Config.PlaneFindingMode.HORIZONTAL`;
   - `Config.LightEstimationMode.ENVIRONMENTAL_HDR`;
   - `Config.DepthMode.AUTOMATIC`, если доступен;
   - `Config.FocusMode.AUTO`;
   - `Config.UpdateMode.LATEST_CAMERA_IMAGE`.
3. На каждом AR-кадре вызывается `viewModel.onSessionUpdated(...)`.
4. Use case передает кадр в repository.
5. Repository возвращает `ArFrameResult`.
6. ViewModel обновляет `FloorUiState`.
7. Compose и SceneView перерисовывают AR-узлы и overlay UI.

## Состояние Экрана

`FloorUiState` хранит:

- состояние поиска пола и tracking;
- количество горизонтальных плоскостей;
- примерную площадь выбранной ARCore-плоскости;
- наличие hit-test в центре экрана;
- включен ли depth;
- статусный и instructional text через `UiText`;
- список точек контура;
- замкнут ли полигон;
- финализирована ли секция;
- выбранный поворот текстуры;
- выбранный тип плитки;
- текущий `HitResult`/`Pose`;
- индекс точки, к которой сейчас происходит snap.

Точки хранятся в UI state, а не в repository. Это позволяет ViewModel явно управлять пользовательским сценарием и не смешивать AR-данные с UI-логикой.

## Разметка Контура

Точки добавляются по текущему hit-test результату из центра экрана. Для стабилизации используется ARCore `Anchor`: при добавлении точки ViewModel вызывает `hitResult.createAnchor()`, а затем сохраняет `ArPoint(anchor, anchor.pose)`.

На каждом AR-кадре ViewModel обновляет pose всех точек:

```kotlin
val updatedPoints = currentState.points.map { point ->
    point.copy(pose = point.anchor.pose)
}
```

Так Compose получает новое состояние и корректно перерисовывает контур после изменений tracking, включая переход приложения в `onStop` и возврат обратно.

## Ограничения При Добавлении Точек

ViewModel не добавляет новую точку, если:

- секция уже финализирована;
- reticle сейчас snapped к существующей точке;
- новая точка слишком близко к последней;
- точка сильно отличается по высоте от первой точки контура;
- ARCore не дал валидный `HitResult` или не удалось создать `Anchor`.

Это уменьшает случайные дубли, скачки по высоте и нестабильные точки.

## Замыкание Полигона

ViewModel проверяет расстояние от текущего reticle hit pose до первой точки:

- если точек минимум 3 и расстояние меньше `CLOSE_THRESHOLD_M`, полигон считается замкнутым;
- если расстояние меньше `SNAP_THRESHOLD_M`, подсвечивается точка snapping;
- при замкнутом полигоне кнопка добавления работает как `OK`;
- нажатие `OK` переводит секцию в `isFinalized = true`.

Важно: совпадение первой и последней точки само по себе не включает текстуру плитки. До нажатия `OK` контур только считается замкнутым и показывается обычной заливкой.

## Финализация

После финализации:

- добавление и undo точек блокируются;
- ARCore plane renderer скрывается, чтобы не мешать просмотру секции;
- внутри полигона отображается выбранная текстура плитки;
- становятся доступны действия с плиткой: поворот узора и смена типа плитки.

Сброс секции отсоединяет anchors, очищает точки, снимает финализацию и возвращает поворот текстуры к `TextureRotation.DEGREES_0`.

## Визуализация

Модуль отображает:

- зеленые плоские точки через `CylinderNode`;
- синие линии через `CubeNode`;
- динамическую линию от последней точки до reticle;
- подписи длин через `ImageNode` с bitmap-лейблами;
- заливку полигона через `ShapeNode`;
- текстуру плитки после финализации.

Визуальный порядок:

1. заливка или текстура;
2. линии;
3. подписи;
4. точки.

Так точки остаются поверх линий и заливки.

## Текстуры Плитки

Текстуры хранятся в:

```text
src/main/assets/textures/
```

Сейчас доступны:

- `paving_stones.png`;
- `paving_stones_v2.png`.

Обе текстуры считаются плиткой одного физического размера:

- ширина: `0.78 м`;
- высота: `1.04 м`.

Выбор текстуры хранится в enum:

```kotlin
enum class TileType(val assetPath: String) {
    CLASSIC("textures/paving_stones.png"),
    MODERN("textures/paving_stones_v2.png")
}
```

Сейчас по умолчанию выбран `TileType.MODERN`.

## Поворот Текстуры

Поворот узора хранится как enum:

```kotlin
enum class TextureRotation {
    DEGREES_0,
    DEGREES_45,
    DEGREES_90,
    DEGREES_135
}
```

В доменной модели не хранится raw `Float` с градусами. Циклический переход выполняет ViewModel через `TextureRotation.entries` и `ordinal`, а маппинг enum в градусы находится на UI/rendering уровне.

Это удобно для будущего backend-контракта: на сервер можно отправлять стабильное enum-значение, а не произвольный float.

`DEGREES_0` означает не глобальный world-axis и не произвольный угол bitmap, а базовую раскладку вдоль длинной стороны пользовательской секции. Пользовательские повороты `45/90/135°` применяются относительно этой базовой ориентации.

## Как Работает Текстурирование

После финализации `ShapeNode` использует текстурированный материал. До финализации замкнутый полигон отображается обычной цветной заливкой.

Для корректного `0°` используется локальная система координат секции:

1. По точкам полигона находится самая длинная сторона.
2. Ось `X` локального `ShapeNode` строится вдоль этой длинной стороны.
3. Ось `Y` строится перпендикулярно ей.
4. Точки полигона проецируются в эту локальную систему координат.
5. Сам `ShapeNode` поворачивается в AR-мире через `rotationY`, чтобы геометрия осталась на реальных anchor-точках.
6. Для каждого `TextureRotation` создается bitmap-паттерн, где `BitmapShader` повторяет плитку с учетом физического размера `780x1040 мм`.
7. `DEGREES_0` ложится вдоль локальной оси `X`, а значит вдоль длинной стороны секции.

Это важно: пользовательская секция фиксирована по anchor-точкам, не сжимается в ромб и не зависит от глобальных осей ARCore. Вращается только раскладка плитки относительно корректного локального нуля.

## Границы Ответственности

В модуль можно добавлять:

- новые AR-визуализации выделенной зоны;
- новые состояния разметки;
- UI выбора плитки, угла поворота и параметров примерки;
- сетевой use case отправки выделенного контура на backend;
- маппинг AR-точек в 2D backend DTO;
- экспорт `TileType` и `TextureRotation`.

Не стоит добавлять:

- расчет оптимальной раскладки плитки;
- работу с базой данных проектов;
- хранение каталога плитки на сервере;
- тяжелую оптимизацию обрезков;
- backend-specific бизнес-логику.

Эти задачи должны перейти в отдельный backend-проект, описанный в `../../BACKEND_ROADMAP.md`.

## Добавление Новой Плитки

Чтобы добавить новый вариант плитки:

1. Положить изображение в `src/main/assets/textures/`.
2. Добавить новый элемент в `TileType`.
3. Убедиться, что для новой картинки известны физические размеры.
4. Если размер отличается от `780x1040 мм`, вынести размеры из констант в модель плитки.

Сейчас обе имеющиеся текстуры имеют одинаковые физические размеры, поэтому используют общие настройки.

## Ближайшее Развитие

Следующий логичный шаг для этого модуля - подготовить экспорт выделенной зоны:

- преобразовать AR `x/z` координаты в 2D-полигон в метрах;
- нормализовать точки относительно первой точки или центра;
- добавить выбранный `TileType`, `TextureRotation` и физический размер плитки;
- отправить данные в backend через отдельный repository/use case.
